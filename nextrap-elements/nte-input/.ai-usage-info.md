# @nextrap/nte-input

## Purpose

Minimal wrapper component for input controls.

`nte-input` only renders the shared frame:
- label
- control container
- validation area

The concrete control markup and setup come from statically registered plugins.

Built-in plugins currently include:
- text / email / password
- textarea
- select
- select-radio
- checkbox

## Plugin architecture

A plugin may register multiple input types.
A single input type may only be registered by exactly one plugin.

```ts
export interface NteInputPlugin {
  types: string[];
  getHtml?: (context: unknown) => unknown;
  init?: (element: NteInput) => void;
  shouldHoverlabelFloat?: (element: NteInput) => boolean;
}
```

Registration:

```ts
NteInput.registerPlugin({
  types: ['select', 'multiselect', 'autocomplete'],
  getHtml: (context) => html`...`,
  init: (element) => {},
});
```

## Render flow

1. Plugin is registered in TypeScript.
2. `registerPlugin()` stores each type separately in the static registry.
3. During render, `nte-input` resolves the plugin by `type`.
4. If found, `getHtml()` is used.
5. If present, `init()` runs after render.

## Available base API

Attributes / properties commonly used by plugins:

- `type`
- `label`
- `validation-message`
- `invalid`
- `data-options`
- `value`

Context helper exported by the package:

```ts
interface NteInputRenderContext {
  element: NteInput;
  type: string;
  controlId: string;
  validationId: string;
}
```

## Built-in textarea plugin behavior

The default `textarea` plugin behaves like this:

- renders a native `<textarea>` inside the shared frame
- keeps the textarea height in sync with its content
- clamps the auto height between CSS min and max height
- syncs the host `value` attribute with the textarea value
- reacts to external `value` attribute changes as well

Relevant CSS variables:
- `--nte-input-textarea-min-height`
- `--nte-input-textarea-max-height`

## Built-in select plugin behavior

The default `select` plugin behaves like this:

- if parsed `data-options` exist, it uses them as the option source
- otherwise it looks for an `<options>` element inside `<nte-input>`
- on init, it copies those `<option>` entries into the internal `<select>`
- after syncing options, it applies the current `value` attribute to select the matching option

`data-options` supports:
- semicolon-separated strings like `draft|Entwurf;active|Aktiv`
- JSON strings

Those values are transformed into `InputOptionsType`.

## Built-in select-radio plugin behavior

The default `select-radio` plugin uses the same option resolution as `select`:

- if parsed `data-options` exist, they win
- otherwise `<options>` inside `<nte-input>` are used
- with `multiple="false"` / omitted, it renders one radio per option
- with `multiple`, it renders one checkbox per option
- it syncs the rendered checks with the host `value` attribute
- on user changes, it writes the selected value back to `value`
  - single mode: plain string
  - multiple mode: JSON array string

## Example plugin

```ts
import { html } from 'lit';
import { NteInput, type NteInputRenderContext } from '@nextrap/nte-input';

NteInput.registerPlugin({
  types: ['text', 'email', 'password'],
  getHtml: (context) => {
    const { element, controlId, validationId } = context as NteInputRenderContext;

    return html`
      <input
        id=${controlId}
        type=${element.type}
        placeholder=${element.getAttribute('placeholder') ?? ''}
        aria-describedby=${validationId}
      />
    `;
  },
});
```

## Styling mixins

The package also exposes a root `mixins.scss` file.

Available mixin:

```scss
@use '@nextrap/nte-input/mixins' as nte-input;

nte-input.hoverlabel {
  @include nte-input.hoverlabel;
}
```

`hoverlabel` enables a Bootstrap-like floating label using the component parts and the reflected `has-value`, `has-placeholder` and `hoverlabel-active` states.
It also increases the control height / inner spacing so the label has enough room across browsers.
Plugins can force the floated state via `shouldHoverlabelFloat()`.

## Agent hints

- Keep `nte-input` itself minimal. Do not add type-specific logic to the base component unless explicitly requested.
- Prefer plugins for every concrete input variant.
- If a new input type is needed, extend via `registerPlugin()` instead of growing the base element.
- `checkbox` is a special case: its text label is rendered by the plugin next to the checkbox, and the normal framed control styling is removed via CSS for `type="checkbox"`.
- Keep the public theming API on `:host` CSS variables.
