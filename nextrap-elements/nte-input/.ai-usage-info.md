# @nextrap/nte-input

## Purpose

Minimal wrapper component for input controls.

`nte-input` only renders the shared frame:
- label
- control container
- validation area
- optional input aid area

The concrete control markup and setup come from statically registered plugins.

Built-in plugins currently include:
- text / email / password
- textarea
- select
- select-radio
- checkbox
- token-input

## Plugin architecture

A plugin may register multiple input types.
A single input type may only be registered by exactly one plugin.

```ts
class MyPlugin extends AbstractNteInputPlugin {
  static readonly types = ['select'];

  render(context: NteInputRenderContext) {
    return html`...`;
  }
}
```

Registration:

```ts
NteInput.registerPlugin(MyPlugin);
```

## Render flow

1. Plugin class is registered in TypeScript.
2. `registerPlugin()` stores each type separately in the static registry.
3. `nte-input` resolves the plugin once by `type`.
4. It creates one plugin instance and keeps it.
5. The plugin instance handles render, lifecycle and value logic.

## Available base API

Attributes / properties commonly used by plugins:

- `type`
- `label`
- `validation-message`
- `invalid`
- `data-options`
- `value`

Named slots on the base component:
- `validation`
- `input-aid`

The host element forwards plugin value APIs:

- `element.value`
- `element.selectedOptions`

`nte-input` is form-associated. Named elements participate in native form submission and `new FormData(form)`.

Built-in behavior:
- text / email / password / textarea: `value` is a string
- select: `value` is a string
- checkbox: `value` is a boolean
- select-radio: `value` is always a `string[]`
- token-input: `value` is always a `string[]`

Context helper exported by the package:

```ts
interface NteInputRenderContext {
  element: NteInput;
  type: string;
  controlId: string;
  validationId: string;
}
```

## Built-in textarea plugin behavior

The default `textarea` plugin behaves like this:

- renders a native `<textarea>` inside the shared frame
- keeps the textarea height in sync with its content
- clamps the auto height between CSS min and max height
- syncs the host `value` property with the textarea value
- reacts to external `value` property changes as well

Relevant theme keys:
- `textarea-min-height`
- `textarea-max-height`

## Built-in select plugin behavior

The default `select` plugin behaves like this:

- if parsed `data-options` exist, it uses them as the option source
- otherwise it looks for an `<options>` element inside `<nte-input>`
- on init, it renders those options into the internal `<select>`
- it marks the option matching the current host `value` as selected

`data-options` supports:
- semicolon-separated strings like `draft|Entwurf;active|Aktiv`
- JSON strings

Those values are transformed into `InputOptionsType`.

## Built-in select-radio plugin behavior

The default `select-radio` plugin uses the same option resolution as `select`:

- if parsed `data-options` exist, they win
- otherwise `<options>` inside `<nte-input>` are used
- with `multiple="false"` / omitted, it renders one radio per option
- with `multiple`, it renders one checkbox per option
- it syncs the rendered checks with the host `value` property
- on user changes, it writes the selected values back to `value`
  - single mode: `string[]` with one item
  - multiple mode: `string[]`

## Built-in token-input plugin behavior

The default `token-input` plugin behaves like this:

- renders selected values as removable tokens inside the shared frame
- keeps the host `value` property as a deduplicated `string[]`
- supports the same option resolution as `select`
  - if parsed `data-options` exist, they win
  - otherwise `<options>` inside `<nte-input>` are used
- uses resolved options as suggestions via a native `<datalist>`
- also allows free custom values not present in the options
- with `strict`, only configured option values may be added
- adds tokens on `Enter`, `,`, `;`, blur or change
- removes tokens via a remove button or `Backspace` when the draft input is empty
- for `required`, validity means: at least one token is selected

Example:

```html
<nte-input
  label="Schlagworte"
  type="token-input"
  value='["news"]'
  data-options='[{"value":"news","label":"News"},{"value":"events","label":"Events"}]'
></nte-input>

<nte-input
  label="Schlagworte (strict)"
  type="token-input"
  strict
  data-options='[{"value":"news","label":"News"},{"value":"events","label":"Events"}]'
></nte-input>
```

## Example plugin

```ts
import { html } from 'lit';
import { NteInput, type NteInputRenderContext } from '@nextrap/nte-input';

NteInput.registerPlugin({
  types: ['text', 'email', 'password'],
  getHtml: (context) => {
    const { element, controlId, validationId } = context as NteInputRenderContext;

    return html`
      <input
        id=${controlId}
        type=${element.type}
        placeholder=${element.getAttribute('placeholder') ?? ''}
        aria-describedby=${validationId}
      />
    `;
  },
});
```

## Validation and input aid bubbles

Below the control, `nte-input` can render two optional bubble-like helper areas:

- `slot="validation"` or `validation-message`
- `slot="input-aid"`

Behavior:
- validation is rendered above input aid when both exist
- validation uses the danger color palette
- input aid is only expanded while the component is focused (`:focus-within`)
- input aid visibility in CSS relies on the `slot-empty` class applied by `nte_element`
- validation visibility is driven by either a non-empty `validation-message` attribute or a non-empty `validation` slot

Example:

```html
<nte-input
  label="Passwort"
  type="password"
  invalid
  validation-message="Bitte mindestens 12 Zeichen eingeben."
>
  <span slot="input-aid">Nutze Groß-/Kleinbuchstaben, Zahlen und Sonderzeichen.</span>
</nte-input>
```

## Styling mixins

The package exposes styling mixins from `src/styles/*`.

Available mixins:

```scss
@use '@nextrap/nte-input/src/styles/theme';
@use '@nextrap/nte-input/src/styles/floating-labels';
@use '@nextrap/nte-input/src/styles/select-radio-vertical';
@use '@nextrap/nte-input/src/styles/size';

nte-input.default {
  @include theme.theme(theme.$themeBootstrap);
}

nte-input.carbon {
  @include theme.theme(theme.$themeCarbon);
}

nte-input.default.hoverlabel,
nte-input.carbon.hoverlabel {
  @include floating-labels.floating-labels();
}

.default.select-radio-vertical {
  nte-input[type='select-radio'] {
    @include select-radio-vertical.select-radio-vertical();
  }
}
```

`theme()` styles the shared frame, helper bubbles, checkbox/select-radio layout and token presentation based on a theme map.
The package exports `theme.$themeDefaults`, `theme.$themeBootstrap` and `theme.$themeCarbon`.
Prefer extending the exported theme maps instead of growing Shadow DOM CSS.

`theme.$themeCarbon` provides a Carbon-like alternative theme with square controls, neutral filled backgrounds and a stronger bottom-border focus treatment.
It uses the same shared scale variables as the other style mixins.

`floating-labels()` enables a Bootstrap-like floating label using the component parts and the reflected `has-value`, `has-placeholder` and `hoverlabel-active` states.
It increases the control height / spacing from the `$minHeight` SCSS argument.
Plugins can force the floated state via `isHoverlabelActive()`.

`select-radio-vertical()` is intended for `type="select-radio"` and should usually be applied from a form or container selector.
It styles only descendant `nte-input[type="select-radio"]` elements and lays out their options side by side inside the input.
It uses a minimum column width of `350px`, shows a separator between columns and takes SCSS arguments with defaults.

`size()` is a small helper mixin that sets the shared CSS scale variables used by `theme()` and `floating-labels()`.
It accepts separate scales for spacing, label and input.
The bundled `src/styles/index.scss` already defines example classes: `sm`, `md`, `lg`, `xl`, `xxl`.

```scss
@use '@nextrap/nte-input/src/styles/size';

nte-input.sm {
  @include size.size(0.875, 0.875, 0.875);
}

nte-input.xxl {
  @include size.size(1.35, 1.15, 1.15);
}
```

## Agent hints

- Keep `nte-input` itself minimal. Do not add type-specific logic to the base component unless explicitly requested.
- Prefer plugins for every concrete input variant.
- If a new input type is needed, extend via `registerPlugin()` instead of growing the base element.
- Keep logic in plugin classes, not in `nte-input`.
- Plugin-specific Shadow DOM CSS belongs next to the plugin in `src/plugins/*.scss` and is returned via `getStyleSheet()` so `nte-input` can inject it into the shared shadow root.
- Keep Shadow DOM CSS minimal and functional; visual styling should usually live in `src/styles/*.scss` mixins via `::part(...)`.
- `checkbox` is a special case: its text label is rendered by the plugin next to the checkbox, and the default frame should stay disabled for `type="checkbox"` in the light DOM theme.
- `token-input` should keep Shadow DOM styling minimal and expose the important inner elements via `part` attributes (`token-list`, `token`, `token-text`, `token-remove`, `token-input`).
- Prefer SCSS mixins and existing global `--nt-*` variables over introducing new component-specific CSS variables.
