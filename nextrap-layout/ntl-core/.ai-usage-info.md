# @nextrap/ntl-core

## Purpose

Core factory for all `ntl-*` layout web components.

Use this package when creating layout-level Nextrap components that build on `@nextrap/nte-core` and optionally apply nested layout behavior.

## Construction rule

For AI-generated code and package conventions:
- all `ntl-*` layout elements should be constructed via `ntl-element()`
- in the current codebase this corresponds to the factory exported by `@nextrap/ntl-core`, currently `nextrap_layout()`

So when editing existing packages, use the real export name from the package, but keep the rule intact: all layout components go through the shared `ntl-element()`/`nextrap_layout()` base factory.

## Import

```ts
import { nextrap_layout } from '@nextrap/ntl-core';
```

## Relationship to `@nextrap/nte-core`

`@nextrap/ntl-core` extends the `nte-core` base factory and therefore also inherits the shared browser-oriented feature set from `@trunkjs/browser-utils`.

Reference package:
- `/opt/node_modules/@trunkjs/browser-utils/.ai-usage-info.md`

Also relevant for nested layout application:
- `@trunkjs/content-pane` / `SubLayoutApplyMixin`

## Supported features

All `nte-core` features are available:

```ts
{
  logging?: boolean;
  slotVisibility?: boolean;
  eventBinding?: boolean;
  breakpoints?: boolean;
}
```

Additionally:

```ts
{
  subLayoutApply?: boolean;
}
```

Current defaults:

```ts
{
  logging: true,
  slotVisibility: false,
  eventBinding: false,
  breakpoints: false,
  subLayoutApply: false,
}
```

## Common examples

### Minimal `ntl-*` layout component

```ts
import { html, unsafeCSS } from 'lit';
import { customElement } from 'lit/decorators.js';
import { nextrap_layout } from '@nextrap/ntl-core';
import { resetStyle } from '@nextrap/style-reset';
import style from './ntl-demo.scss?inline';

@customElement('ntl-demo')
export class NtlDemo extends nextrap_layout() {
  static override styles = [unsafeCSS(resetStyle), unsafeCSS(style)];

  override render() {
    return html`<slot></slot>`;
  }
}
```

### Typical layout with nested content processing

```ts
import { html, unsafeCSS } from 'lit';
import { customElement } from 'lit/decorators.js';
import { nextrap_layout } from '@nextrap/ntl-core';
import { resetStyle } from '@nextrap/style-reset';
import style from './ntl-2col.scss?inline';

@customElement('ntl-2col')
export class Ntl2Col extends nextrap_layout({
  breakpoints: true,
  subLayoutApply: true,
  slotVisibility: true,
}) {
  static override styles = [unsafeCSS(resetStyle), unsafeCSS(style)];

  override render() {
    return html`
      <div id="container" part="container">
        <slot></slot>
        <slot name="aside" data-query=":scope > .aside"></slot>
      </div>
    `;
  }
}
```

### Layout with browser-utils powered event bindings

```ts
import { html } from 'lit';
import { customElement } from 'lit/decorators.js';
import { Listen } from '@trunkjs/browser-utils';
import { nextrap_layout } from '@nextrap/ntl-core';

@customElement('ntl-observer')
export class NtlObserver extends nextrap_layout({ eventBinding: true }) {
  override render() {
    return html`<slot></slot>`;
  }

  @Listen('resize', { target: 'window' })
  onResize() {
    this.requestUpdate();
  }
}
```

## Agent hints

- For every new `ntl-*` component, start from `nextrap_layout(...)` instead of `LitElement` directly.
- If the component restructures, re-slots, or applies nested layout behavior, consider `subLayoutApply: true`.
- Reuse `@trunkjs/browser-utils` behavior through the existing core feature flags before introducing custom logic.
- Keep diffs small and follow existing `ntl-*` component patterns such as `ntl-2col`, `ntl-card`, and `ntl-hero`.
- Use `@nextrap/style-reset` for shadow DOM baseline styles in layout components.
